# AI_REFLECTION.md — Рефлексия по использованию AI-инструментов

## 3.1. Применённые инструменты и промпты

### Инструменты

В ходе разработки использовались три AI-инструмента с чёткими ролями:

**Qwen Code** — основной инструмент разработки. Использовался для генерации всего кода проекта через команды Spec Kit (`/speckit.specify`, `/speckit.plan`, `/speckit.tasks`, `/speckit.implement`). Qwen Code работал с полным контекстом проекта и генерировал файлы сразу в нужных директориях.

**Windsurf** — резервный инструмент, использовался когда Qwen Code не справлялся с задачей или давал неудачный результат. Особенно полезен был при сложных багах в frontend state management и рефакторинге архитектуры.

**Claude (claude.ai)** — использовался для проектирования архитектуры до написания кода, принятия технических решений (выбор стека, SQLite vs PostgreSQL, подход к авторизации), дебаггинга сложных ошибок по логам, написания документации (`design.md`, `AI_REFLECTION.md`) и формулировки промптов для Spec Kit.

### Сводная таблица инструментов

| Задача | Инструмент |
|--------|------------|
| Архитектура, `design.md`, спецификации | Claude |
| Генерация кода через Spec Kit | Qwen Code |
| Сложные баги, резервный инструмент | Windsurf |
| `AI_REFLECTION.md`, документация | Claude |

### Примеры конкретных промптов

**Проектирование (Claude):**
```
/speckit.specify Build an interactive geography quiz web application. Users see a question
about a location and must click on the correct place on a world map. The system calculates
the distance between the click and the correct answer, awards points based on accuracy
and speed...
```
Цель: сформулировать требования до написания кода, не привязываясь к стеку.

**Технический план (Qwen Code):**
```
/speckit.plan Use React + Vite + TypeScript for frontend with react-leaflet for the
interactive map. Backend is Python FastAPI with SQLite database via SQLAlchemy. Scoring
is calculated server-side based on Haversine distance and time remaining. No authentication
required — before starting a game, user enters their name (required, max 20 chars)...
```
Цель: задать конкретный стек и ключевые архитектурные решения.

**Дебаггинг (Claude → Windsurf):**
```
Fix two bugs in the backend:
1. In src/models/answer.py - CheckConstraint is not imported.
2. In src/services/scoring.py line 6 - duplicate argument 'lat2'. The last parameter
should be 'lon2', not 'lat2'.
```
Цель: точечное исправление конкретных ошибок с указанием файла и строки.

**Рефакторинг (Qwen Code):**
```
Perform a code quality refactor applying SOLID principles:
SRP: routes.py must only handle HTTP request/response, no business logic.
OCP: ScoringService should be extensible for new scoring strategies.
DIP: Services should depend on abstractions, not concrete implementations.
Add structlog throughout...
```
Цель: улучшить архитектуру после завершения функциональной разработки.

## 3.2. Анализ эффективности

### Где AI ускорил разработку

**Генерация boilerplate кода** — наибольший выигрыш. Настройка FastAPI с CORS, SQLAlchemy моделями, Pydantic схемами, React компонентами с TypeScript типами — всё это заняло бы значительно больше времени вручную. Spec Kit позволил сгенерировать 24 файла за один `/speckit.implement`.

**Проектирование архитектуры** — Claude помог принять обоснованные решения до написания кода: отказ от PostgreSQL в пользу SQLite, отказ от авторизации в пользу имени + UUID, выбор Leaflet вместо Google Maps. Это сэкономило время на переработку.

**Дебаггинг по логам** — когда бэкенд возвращал 500 или 422 ошибки, достаточно было вставить traceback в чат и получить точное указание на проблему с объяснением причины.

**Написание документации** — `design.md` с реальными сниппетами кода, таблицами моделей и примерами API был сгенерирован Qwen Code за один промпт с доступом к исходному коду.

### Где AI ошибся

**Дублирование аргумента `lat2`** — в функции `haversine_distance` последний параметр был назван `lat2` вместо `lon2`. Ошибка обнаружена при первом запуске бэкенда через traceback Python. Причина: AI не всегда проверяет уникальность имён аргументов при генерации математических функций.

**Синтаксические ошибки в `routes.py`** — после рефакторинга бэкенд падал с `SyntaxError: parameter without a default follows parameter with a default`. AI неправильно упорядочил параметры FastAPI функции — параметр с `Depends()` шёл перед обязательными параметрами. Ошибка типичная для тех кто не знает правил Python о порядке аргументов.

**Неверный подсчёт `round` во фронтенде** — после генерации MVP состояние `round` оставалось `undefined` при отправке ответа. AI создал `useEffect` с неправильным условием инициализации, из-за чего раунд не создавался. Потребовалось несколько итераций исправлений с добавлением `console.log` для диагностики.

**Ошибка `.count()` в лидерборде** — при расчёте ранка использовался метод `.count()` на объекте `ScalarResult`, которого не существует в SQLAlchemy 2.0. Ошибка типична для AI обученного на коде с более старыми версиями библиотек.

**Карта не отображалась после рефакторинга** — при добавлении хедера с прогрессом и переработке layout карта Leaflet перестала рендериться. Причина: Leaflet требует явной высоты контейнера, которую AI не учёл при изменении CSS. Обнаружено визуально при тестировании.

**Отсутствие логирования на старте** — Qwen Code изначально использовал `print()` по всему бэкенду вместо структурированного логирования. Несмотря на то что structlog был указан в плане, первые фазы реализации (Phase 1-2) были сгенерированы без какого-либо логирования вообще, а где оно появлялось — через `print()`. Потребовался отдельный рефакторинг для замены всех `print()` на structlog с правильной конфигурацией (консоль + JSON файл).

**Нарушение SRP при первой генерации** — до рефакторинга `routes.py` содержал бизнес-логику напрямую: создание сервисов внутри хендлеров, валидация данных в теле функции, SQL-запросы для получения следующего вопроса. Только после промпта с принципами SOLID была проведена переработка: бизнес-логика перемещена в сервисы (`GameService`, `ScoringService`), маршруты оставлены только для HTTP.

**Дублирование валидации** — в нескольких местах (`routes.py`, `services/game.py`) повторялась логика проверки имени игрока и координат. Только после промпта с принципом DRY была создана `utils/validators.py` с централизованной валидацией.

### Сознательный отказ от предложений AI

**SQLite вместо PostgreSQL** — Qwen Code в плане предложил PostgreSQL как более "правильное" решение. Это было отклонено в пользу SQLite, так как для учебного проекта важна простота запуска без установки сервера БД — что прямо указано в требованиях задания.

**Название модуля `logger.py` вместо `logging_config.py`** — Qwen Code создал файл с именем `logging_config.py`. Это было исправлено на `logger.py` как более лаконичное и понятное название для модуля логирования.

**Добавление имени игрока перед игрой** — в изначальном плане авторизация отсутствовала полностью, без какой-либо идентификации. Было принято решение добавить поле ввода имени — это сделало лидерборд осмысленным и улучшило UX, при этом не усложнив архитектуру.

## 3.3. Выводы и рекомендации

### Вывод 1: AI — инструмент для ускорения, а не замена инженерного мышления

Наибольшая польза от AI — на этапах где много механической работы: boilerplate, типовые паттерны, документация. Там где нужно принимать архитектурные решения (какую БД выбрать, как организовать состояние, какие принципы применить) — AI даёт варианты, но финальное решение всегда остаётся за разработчиком. Слепое следование предложениям AI в этих случаях привело бы к избыточной сложности (PostgreSQL для учебного проекта) или неудобному API.

### Вывод 2: Чем точнее промпт — тем лучше результат

Разница между "fix the bug" и "Fix SyntaxError in src/api/routes.py line 43: parameter without a default follows parameter with a default. Move the db parameter to be the last parameter" — это разница между несколькими итерациями и одним точным исправлением. Инвестиция в формулировку промпта окупается сразу.

### Вывод 3: AI генерирует код под старые версии библиотек

Несколько ошибок (.count() в SQLAlchemy 2.0, порядок аргументов в FastAPI) были вызваны тем, что AI обучен на коде с устаревшими API. При использовании современных версий библиотек стоит всегда проверять сгенерированный код на соответствие актуальной документации, особенно в местах работы с БД и фреймворком.

### Как планирую использовать AI в будущих проектах

Использовать связку: Claude для проектирования и принятия решений → Spec Kit + Qwen Code для генерации кода → Claude для дебаггинга сложных проблем и документации, Windsurf для резервного дебаггинга. Фиксировать промпты по ходу работы — это упрощает написание рефлексии и помогает улучшать подход к формулировкам.

## 3.4. Разработка панели администратора

### Добавленный функционал

**Qwen Code** использовался для реализации функционала администрирования вопросов:
- Создание новых вопросов через UI с интерактивной картой
- Редактирование существующих вопросов (текст, координаты, категория, подсказка)
- Удаление вопросов из базы данных
- Авто-расчёт `difficulty` и `location_type` на основе `time_limit` и `category`

### Примеры промптов

**Расширение функционала редактирования:**
```
Extend the edit question dialog to allow editing:
1. Question text (input field)
2. Coordinates (lat/lon inputs + map click to update)
3. Category (dropdown: Countries/Cities/Landmarks/Capitals)
4. Time limit (dropdown: 30/45/60 seconds)
5. Hint (optional input)
Auto-calculate difficulty from time_limit and location_type from category.
```

**Добавление создания вопросов:**
```
Add a "Create Question" button to the admin panel that opens a dialog
similar to the edit dialog. Include a map where admin can click to set
coordinates. Use the same auto-calculation logic for difficulty and location_type.
```

### Выводы

**Исправление багов:** Точечные промпты с указанием файла, строки и ожидаемого поведения работают эффективнее общих описаний проблемы.  

**Паттерн "копировать и адаптировать" работает хорошо**: диалог создания вопросов был быстро реализован на основе существующего диалога редактирования, что обеспечило консистентность UI и сократило время разработки.

### Рекомендации коллегам

Начинайте с проектирования, а не с кода. Потратьте время на описание что вы строите и почему — AI генерирует значительно лучший код когда получает чёткий контекст через спецификацию. Инструменты вроде Spec Kit формализуют этот подход.

Не доверяйте первому результату в местах где важна корректность: математические формулы, работа с БД, управление состоянием. Всегда запускайте и тестируйте. AI уверенно пишет неправильный код.
